# Honcho-Panel — Internationalization (i18n)

> Honcho-Panel supports multiple languages via `react-i18next`. All UI text is extracted into JSON translation files organised by namespace.

## Architecture

```
Browser Language → i18next-browser-languagedetector → i18next
                                                         │
                                                         ├─ en/common.json
                                    useTranslation()  ←  ├─ en/overview.json
                                                         │  ...
                                                         │
                                   LanguageSwitcher  →  localStorage (persist)
```

## Tech Stack

| Component | Choice |
|-----------|--------|
| Core | `i18next` |
| React bindings | `react-i18next` |
| Language detection | `i18next-browser-languagedetector` |
| Translation format | JSON (namespace) |
| Persistence | `localStorage` (`i18nextLng` key) |
| Default language | `en` (also serves as fallback) |
| Plural handling | `i18next` built-in (ICU message format) |

## Supported Languages

| Language | Code | Status |
|----------|------|--------|
| English | `en` | Default, fallback |
| 简体中文 | `zh-CN` | Supported |

Adding a new language requires only creating the corresponding JSON files under `src/i18n/locales/{code}/` and registering it in the i18next configuration.

## Namespace Design

Each namespace maps to a logical domain. Components import only the namespace they need.

| Namespace | Scope | Used By |
|-----------|-------|---------|
| `common` | Sidebar navigation, shared buttons, table column headers, empty state, LangSwitcher | All pages |
| `overview` | Stat card labels, recent sessions section | OverviewPage |
| `peers` | Peer table, peer detail header, tabs (Repr/Card/Chat), chat input | PeerListPage, PeerReprPage, PeerCardPage, PeerChatPage |
| `sessions` | Session table, session view header, message timeline | SessionListPage, SessionViewPage |
| `conclusions` | Conclusion cards, search placeholder, confidence labels | ConclusionsPage |
| `settings` | API URL input, connection status, about section | SettingsPage |

## Usage

### In Components

```tsx
import { useTranslation } from 'react-i18next';

function MyComponent() {
  const { t } = useTranslation('common');

  return (
    <Button>{t('button.save')}</Button>
  );
}
```

### Multiple Namespaces

```tsx
const { t } = useTranslation(['common', 'peers']);
// t('button.save')                    → common
// t('peers:table.column.peer')        → peers (explicit namespace)
```

### Interpolation

```json
{ "connected": "Connected to Honcho v{{version}}" }
```

```tsx
t('status.connected', { version: '3.0.6' });
// "Connected to Honcho v3.0.6"
```

## Language Detection

Detection order (configured in `i18next-browser-languagedetector`):

1. `localStorage` key `i18nextLng` (persisted user choice)
2. `navigator.language` (browser default)
3. Fallback: `en`

Users change language via the `LanguageSwitcher` component in Settings. The choice is stored in `localStorage` and survives page reloads.

## LanguageSwitcher Component

Placed on the Settings page. Renders as a dropdown or segmented control with available languages.

```
English  |  中文
```

When the user selects a language, `i18next.changeLanguage(code)` is called, triggering a re-render of all `useTranslation` hooks.

## Adding a New Language

1. Create directory `src/i18n/locales/{code}/`
2. Copy JSON files from `en/` to the new directory
3. Translate all string values (do NOT translate keys)
4. Register the language in `src/i18n/index.ts` under `supportedLngs`
5. Add the language option to `LanguageSwitcher`
6. Run the app and verify

## Adding New Translation Keys

1. Add the key to **all** language files under the appropriate namespace
2. If the English value is the fallback, add it only in `en/`; keys missing in other languages will fall back to `en`
3. Use the key in component via `t()` with the correct namespace
4. Never hardcode user-visible strings in components

## Type Safety

`src/i18n/types.ts` exports TypeScript types for translation keys, enabling autocompletion and compile-time validation. Types are manually maintained in sync with the JSON files.

```ts
// types.ts
export interface CommonTranslations {
  sidebar: {
    brand: string;
    overview: string;
    peers: string;
    sessions: string;
    conclusions: string;
    settings: string;
  };
  button: {
    save: string;
    cancel: string;
    delete: string;
    create: string;
    send: string;
    test: string;
  };
}
```

When adding new keys, update the type definitions alongside the JSON files.

## Configuration

```ts
// src/i18n/index.ts
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';
import LanguageDetector from 'i18next-browser-languagedetector';

import common_en from './locales/en/common.json';
import overview_en from './locales/en/overview.json';
import peers_en from './locales/en/peers.json';
import sessions_en from './locales/en/sessions.json';
import conclusions_en from './locales/en/conclusions.json';
import settings_en from './locales/en/settings.json';

import common_zh from './locales/zh-CN/common.json';
// ...other zh imports

i18n
  .use(LanguageDetector)
  .use(initReactI18next)
  .init({
    fallbackLng: 'en',
    supportedLngs: ['en', 'zh-CN'],
    defaultNS: 'common',
    ns: ['common', 'overview', 'peers', 'sessions', 'conclusions', 'settings'],
    resources: {
      en: { common: common_en, overview: overview_en, peers: peers_en,
            sessions: sessions_en, conclusions: conclusions_en, settings: settings_en },
      'zh-CN': { common: common_zh, /* ... */ },
    },
    interpolation: {
      escapeValue: false, // React already escapes
    },
    detection: {
      order: ['localStorage', 'navigator'],
      caches: ['localStorage'],
    },
  });

export default i18n;
```

## Scope

Text excluded from i18n (remains in source as-is):

- Log output, error stack traces
- API response data (peer names, session IDs, message content — these are user data, not UI)
- Console messages for debugging
- CSS class names and DOM attributes
